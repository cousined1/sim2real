It tells Augment to do three things in one pass: **audit and harden security**, **verify product functionality with QA**, and **leave the app ready for production release**. OWASP ASVS is designed as a basis for testing web application technical security controls, and OWASP’s prompt-injection guidance recommends input/output validation, structured prompts, least privilege, logging, and human approval for high-risk actions. ## Combined Augment prompt

You are a **principal security engineer, senior QA lead, and production-readiness reviewer**.  
Audit, harden, test, and prepare my SaaS platform **Sim2Real** for production.

Your assignment is to:
1. **Perform a deep security audit**
2. **Fix or harden vulnerabilities where practical**
3. **Run a full QA and regression pass on critical user flows**
4. **Review production-readiness gaps**
5. **Leave the application in a safer, more stable, more deployable state**
6. **Provide a final release-readiness report with pass/fail status**

Treat this as a real B2B SaaS product that may contain:
- frontend app
- backend APIs
- auth/session flows
- database access
- file uploads
- AI/LLM features
- prompt handling
- retrieval or agent workflows
- Stripe or billing logic
- email flows
- admin pages
- multi-tenant data boundaries
- customer telemetry and operational data

Use **OWASP ASVS-style security verification**, **OWASP prompt-injection/LLM defense guidance**, and a **release-readiness QA gate** as your baseline. OWASP ASVS can be used to establish confidence in web application security controls, while release-readiness QA for SaaS commonly includes functional, regression, API, integration, performance, environment, and go/no-go validation. [amwebtech](https://amwebtech.com/services/release-readiness-qa-for-saas/)

## Required workflow

Follow this sequence:

### Phase 1 — Discover
Map the full app:
- frameworks and runtimes
- package managers
- routes and API endpoints
- auth/session model
- RBAC/tenant model
- DB access layer
- file upload/storage paths
- billing/payment integrations
- LLM/AI/prompt workflows
- external service integrations
- background jobs
- env/config loading
- test setup
- deployment assumptions

Create a short internal risk model before editing anything.

### Phase 2 — Security audit
Review the codebase for common and high-risk issues.

Check at minimum:

#### Web app security
- broken auth
- broken authorization / IDOR
- cross-tenant access
- session fixation
- missing secure cookie flags
- missing CSRF protection
- XSS
- SQL/NoSQL injection
- SSRF
- path traversal
- command injection
- open redirects
- CORS misconfiguration
- insecure debug endpoints
- verbose errors
- insecure password reset/account recovery
- missing rate limits
- unsafe file uploads
- missing security headers
- secrets in repo or code
- secrets in logs or error messages
- unsafe admin endpoints
- weak input validation
- unsafe output encoding

#### LLM / prompt-injection / agent risks
If the app uses LLMs, AI features, agents, retrieval, prompt templates, or tool-calling:
- prompt injection
- indirect prompt injection from retrieved docs or uploads
- system prompt exposure
- hidden policy leakage
- chain-of-thought extraction attempts
- unsafe tool invocation
- untrusted content treated as instructions
- insecure output handling
- model-controlled permissions
- unsafe autonomous actions
- secret exfiltration attempts
- HTML/Markdown output injection
- lack of approval gates for dangerous actions

OWASP’s LLM prompt-injection guidance recommends clearly separating instructions from untrusted content, filtering inputs and outputs, enforcing least privilege, logging LLM interactions, and using human approval for privileged or high-risk actions. [genai.owasp](https://genai.owasp.org/llmrisk2023-24/llm01-24-prompt-injection/)

### Phase 3 — Apply fixes and hardening
Implement safe, minimal, production-grade fixes.

#### Auth and access control hardening
- enforce server-side authorization checks everywhere
- validate tenant boundaries on every read/write
- ensure admin-only actions cannot be reached by standard users
- harden password reset and account recovery
- verify session rotation and invalidation

#### Browser and API hardening
- set secure cookie attributes
- add CSRF defenses where needed
- apply security headers, including CSP and X-Content-Type-Options where appropriate
- tighten CORS
- disable caching of authenticated sensitive responses
- standardize safe error handling
- add rate limits and abuse protection to high-risk endpoints

#### Input/output hardening
- add strict schema validation server-side
- reject unexpected fields
- validate file type, size, extension, and content signature
- sanitize or safely render HTML/Markdown/user content
- escape all untrusted output
- redact sensitive fields from logs and API errors

#### LLM hardening
- separate system prompts from user and retrieved content
- delimit untrusted content clearly
- prevent user input from overriding hidden instructions
- block attempts to reveal internal prompts or secrets
- validate model outputs before rendering or execution
- require confirmation and server-side authorization for sensitive tool use
- deny unknown tools or unsafe tool arguments
- add logging and suspicious-pattern detection for prompt abuse
- add refusal rules for obvious injection attempts
- sanitize retrieved/uploaded content before including it in context where possible

OWASP specifically recommends constrained behavior, expected output validation, input/output filtering, structured prompt formats, and ignoring attempts to alter core instructions. [cheatsheetseries.owasp](https://cheatsheetseries.owasp.org/cheatsheets/LLM_Prompt_Injection_Prevention_Cheat_Sheet.html)

### Phase 4 — QA and regression testing
After hardening, run or add tests for the app’s critical user journeys.

#### Functional QA
Verify:
- pages render correctly
- forms submit correctly
- auth flows work end to end
- session persistence works correctly
- role-based views behave correctly
- APIs return expected responses
- billing actions still function
- file uploads work with valid files
- invalid inputs fail safely
- AI flows still work for normal users
- app does not break after hardening changes

#### Regression testing
Ensure new fixes did not break:
- navigation
- dashboard data loading
- contact/signup flows
- pricing/billing pages
- legal page accessibility
- email triggers if present
- search/filtering
- data creation/edit/delete flows
- admin tasks

#### Security QA
Test:
- unauthorized route access
- cross-tenant access attempts
- XSS payloads
- invalid tokens
- forged requests
- rate-limit behavior
- malicious uploads
- prompt injection attempts
- secret extraction attempts
- tool misuse attempts
- malicious document or retrieved-content instruction attempts

A SaaS release-readiness gate typically includes functional and regression checks, API and integration checks, performance/scalability validation, environment validation, and a go/no-go decision. [qawerk](https://qawerk.com/blog/saas-testing-checklist/)

### Phase 5 — Production-readiness review
Review deployment readiness and improve what you safely can.

Check for:
- required env vars validated at startup
- no hardcoded secrets
- production mode disables debug behavior
- health/readiness checks if applicable
- structured logs without sensitive data
- monitoring or alerting hooks/recommendations
- safe migration assumptions
- dependency audit and patching where possible
- CI recommendations for security and test gating
- release blockers and operational risks
- fallback and rollback notes if relevant

ASVS-aligned implementation strategies often include automating verification in CI/CD, especially for validation, error handling, API security, and scanning-related controls. [beaglesecurity](https://beaglesecurity.com/blog/article/how-to-use-owasp-asvs.html)

## Constraints

- Do not stop at passive recommendations
- Make reasonable fixes directly in code where safe
- Prefer secure minimal diffs over unnecessary rewrites
- Preserve working product behavior
- If a risky issue cannot be auto-fixed safely, document it precisely
- Assume hostile authenticated users, hostile uploaded content, and hostile retrieved content
- Treat user input, retrieved data, external content, and model output as untrusted

## Final instruction

Work in this order:
1. Discover
2. Audit
3. Harden
4. Test
5. Verify production readiness
6. Report clearly

This should be a real secure-code review, real QA pass, and real release-readiness audit — not just linting, not just theoretical notes, and not just a checklist.
